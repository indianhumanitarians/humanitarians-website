# Humanitarians Website Project Handbook

This file is the handover document for the Humanitarians public website and admin system. It explains what the project does, how data moves through it, what Supabase owns today, how to operate the admin panel, and how to ask Codex to migrate the project to Firebase later if Supabase is no longer the preferred backend.

## 1. Project Summary

Humanitarians is a public charity website plus a private admin panel.

The public site shows:

- Homepage impact summary.
- Donate / Join information.
- Public-safe case stories.
- Monthly public reports.
- Zakat and Sadaqah explanation.
- Mentorship information and testimonials.
- Contact and referral links.

The admin panel lets authorized users:

- Add, edit, delete, and export case ledger entries.
- Upload case images to Supabase Storage.
- Add, edit, delete, and export mentorship testimonials.
- Upload testimonial images to Supabase Storage.
- Manage support categories and fund types.
- Manage public site settings such as WhatsApp links, QR images, UPI details, bank details, active donor community, and profile/referral links.
- Invite and manage admins from an owner account.
- View dashboard insights from the private case ledger.

The current source of truth is Supabase. Google Sheets is only a private backup/import workflow using the latest-standard tabs generated by the project scripts.

## 2. Current Tech Stack

Frontend:

- React 19.
- TypeScript.
- Vite.
- React Router.
- Recharts.
- Tailwind CSS v4 plus custom CSS in `src/styles.css`.

Backend:

- Supabase Auth for admin login.
- Supabase Postgres for private and public-safe data.
- Supabase Row Level Security for data protection.
- Supabase Storage for images and QR uploads.
- Supabase Edge Function `invite-admin` for owner-only admin invitations.

Deployment:

- Static frontend deploys well on Netlify.
- Supabase database, Auth, Storage, and Edge Functions are deployed separately.

## 6. Supabase Database Tables

The latest intended schema is in `supabase/schema.sql`.

### `admin_profiles`

Stores which Supabase Auth users can access the admin panel.

Columns:

- `user_id` references `auth.users`.
- `email`.
- `role`: `owner`, `admin`, or `viewer`.
- `created_at`.

Important behavior:

- `owner` can manage admins.
- `admin` can manage data.
- `viewer` is not allowed into the admin panel by frontend logic.

### `site_settings`

Stores editable public website settings.

Examples:

- Active donor community.
- Contact email.
- About/profile document URL.
- Case referral form URL.
- WhatsApp links.
- New Members WhatsApp QR image and storage path.
- UPI names, UPI IDs, and QR images.
- Bank account details.

Public rows have `is_public = true` and are fetched by the public site. Internal storage path rows have `is_public = false`.

### `support_categories`

Admin-managed list for the category dropdown in the case form.

Important fields:

- `name`.
- `display_order`.
- `is_active`.

### `fund_types`

Admin-managed list for the fund type dropdown in the case form.

Important fields:

- `name`.
- `display_order`.
- `is_active`.

### `cases`

Private case ledger table. This is the main source of truth for case data.

Primary key:

- `case_number`, for example `HUM-083`.

Important public/admin fields:

- `reporting_month`.
- `reporting_month_sort`.
- `reporting_month_start`.
- `support_category`.
- `support_description`.
- `fund_source`.
- `zakat_amount`.
- `sadaqah_amount`.
- `other_amount`.
- `total_amount` generated from the three amount fields.
- `public_story_title`.
- `public_location`.
- `public_need_summary`.
- `public_support_summary`.
- `public_outcome_summary`.
- `public_follow_up_summary`.
- `show_in_public_stats`.
- `publish_public_story`.

Private fields:

- `beneficiary_name`.
- `beneficiary_phone`.
- `beneficiary_private_location`.

Audit fields:

- `created_by`.
- `updated_by`.
- `created_at`.
- `updated_at`.

Privacy rule:

Never expose beneficiary name, phone, or private location on the public site.

### `case_images`

Stores image metadata for cases. The actual files live in Supabase Storage.

Important fields:

- `case_number`.
- `display_order`, 1 to 3.
- `storage_path`.
- `public_url`.

The admin form uploads files. Admins should not paste image URLs.

### `mentorship_testimonials`

Private table for mentorship testimonials.

Primary key:

- `testimonial_id`.

Important fields:

- `anonymized_name`.
- `public_role`.
- `mentorship_track`.
- `mentee_stage`.
- `public_location`.
- `period_label`.
- `outcome_summary`.
- `testimonial_text`.
- `profile_image_storage_path`.
- `profile_image_url`.
- `carousel_tagline`.
- `consent_received`.
- `privacy_note`.
- `editing_note`.
- `created_by`.
- `updated_by`.
- `created_at`.
- `updated_at`.

Public rule:

Only testimonials with `consent_received = true` should be visible publicly.

## 7. Supabase Public Views

### `public_case_ledger`

Public-safe view derived from `cases` and `case_images`.

It exposes:

- Public stats rows.
- Published story rows.
- Approximate/safe amounts.
- Public story fields.
- Public image URLs for published stories.

It hides:

- Beneficiary name.
- Phone.
- Private address.
- Private notes.
- Exact sensitive data not needed on the public site.

### `public_case_stories`

Subset of `public_case_ledger` where `publish_public_story = true`.

### `public_mentorship_testimonials`

Public-safe testimonial rows where `consent_received = true`.

## 8. Supabase Storage Buckets

The schema creates these public buckets:

- `case-images`.
- `testimonial-images`.
- `payment-qr-images`.
- `site-setting-images`.

Storage usage:

- Case images go to `case-images`.
- Testimonial profile images go to `testimonial-images`.
- Sadaqah and Zakat QR uploads go to `payment-qr-images`.
- New Members WhatsApp QR uploads go to `site-setting-images`.

Important deletion behavior:

- The admin UI should delete Storage objects when deleting cases/testimonials or replacing QR images.
- Database cascading deletes remove database rows, but Storage files are separate objects and must be deleted through Storage API logic.

## 10. Admin Data Flow

### Login

1. Admin enters email and password on `/admin/login`.
2. `src/services/supabaseAuth.ts` calls Supabase Auth password grant.
3. Supabase returns an access token, refresh token, expiry, and user.
4. The app fetches `admin_profiles` for that user.
5. If the profile is `owner` or `admin`, the session is saved in browser local storage under `humanitarians.admin.session.v1`.
6. Admin routes use `ProtectedAdminRoute`.

### Session Refresh

If the access token is close to expiring, the app uses the refresh token to request a new session from Supabase Auth. The updated session is saved back to local storage.

### Owner Admin Invites

1. Owner opens `/admin/admins`.
2. Owner invites an email and role.
3. Frontend calls `supabase/functions/invite-admin`.
4. Edge Function verifies the current user is an owner.
5. Edge Function creates/invites the Supabase Auth user.
6. Edge Function upserts `admin_profiles`.
7. Invited user opens the email link, lands on `/admin/accept-invite`, and sets a password.

### Admin Case Management

Case form:

- Generates the next `HUM-###` case number.
- Uses a month picker so months normalize to labels such as `Jun 2026`.
- Uses DB-driven category and fund type dropdowns.
- Uploads images to Storage and records them in `case_images`.
- Defaults `show_in_public_stats` to true.
- `publish_public_story` controls whether story fields and images become public.

### Admin Testimonial Management

Testimonial form:

- Uses `testimonial_id` as the primary ID.
- Uploads profile image to Supabase Storage.
- Uses `consent_received` as the public visibility flag.

### Admin Lists and Settings

`/admin/lists` manages:

- Public site settings.
- Support categories.
- Fund types.

Settings are grouped into:

- Impact.
- Contact.
- Payment with Sadaqah, Zakat, and Bank subsections.

QR uploads replace older uploaded QR objects. Reset returns the setting row to the default configured value.

## 11. Scripts

### Create latest-standard Google Sheets CSVs

Script:

```bash
npm run export:supabase-to-sheets
```

This reads Supabase private tables using local `.env`:

```txt
VITE_SUPABASE_URL
SUPABASE_SERVICE_ROLE_KEY
```

Supabase is the only export source because it is the current source of truth. The script reads these private tables directly:

```txt
cases
case_images
mentorship_testimonials
support_categories
fund_types
```

It creates import-ready CSV files in:

```txt
.google-sheets-export/
```

Generated files:

```txt
case-ledger-latest.csv
case-images-latest.csv
mentorship-testimonials-latest.csv
support-categories-latest.csv
fund-types-latest.csv
README.md
```

These files use the latest project column standards, not the old Google Sheet column names.

Recommended Google Sheets import process:

1. Create a blank Google Sheet.
2. Import `case-ledger-latest.csv` first and choose "Replace spreadsheet".
3. Import the remaining CSV files and choose "Insert new sheet(s)".
4. Rename the tabs:
   - `Case Ledger`
   - `Case Images`
   - `Mentorship Testimonials`
   - `Support Categories`
   - `Fund Types`
5. Keep the new Google Sheet private because the case ledger can contain beneficiary names, phone numbers, and private addresses.

The export folder is ignored by Git through `.gitignore` because it may contain private data. Commit the script, not the generated CSV exports.

Optional output folder:

```bash
npm run export:supabase-to-sheets -- --out /path/to/private/export-folder
```

### Import Google Sheets backup into Supabase

Script:

```bash
npm run import:sheets-to-supabase
```

This is the future import path if you manually maintain the private Google Sheets backup. The importer understands the latest-standard backup tabs:

```txt
Case Ledger
Case Images
Mentorship Testimonials
Support Categories
Fund Types
```

Dry run only unless `--write` is provided.

Write:

```bash
npm run import:sheets-to-supabase -- --write
```

Restore one case:

```bash
npm run import:sheets-to-supabase -- --case HUM-001 --write
```

Skip image import:

```bash
npm run import:sheets-to-supabase -- --write --skip-images
```

Required local variables:

```txt
VITE_SUPABASE_URL
SUPABASE_SERVICE_ROLE_KEY
VITE_STATS_CASE_LEDGER_CSV_URL
VITE_STATS_CASE_IMAGES_CSV_URL
VITE_STATS_MENTORSHIP_TESTIMONIALS_CSV_URL
VITE_STATS_SUPPORT_CATEGORIES_CSV_URL
VITE_STATS_FUND_TYPES_CSV_URL
```

The importer expects the new Google Sheets backup generated by `npm run export:supabase-to-sheets`. Previous sheet column names are no longer supported.

### Merge duplicate cases

Script:

```bash
npm run merge:cases -- --target HUM-059 --source HUM-061
```

Dry run is default.

Write:

```bash
npm run merge:cases -- --target HUM-059 --source HUM-061 --write
```

Force fund type and amount column:

```bash
npm run merge:cases -- --target HUM-059 --source HUM-061 --fund-type Sadaqah --amount-field sadaqah_amount --write
```

Behavior:

- Keeps the target case.
- Adds source amounts to target.
- Copies source images into empty target image slots.
- Deletes the source case.
- By default renumbers later `HUM-###` cases down by one.
- Moves affected case image objects into matching Storage folders.
- Writes local backups in `.merge-backups/`.

Always run dry run first.

## 13. Recovery and Backup Plan

If Supabase has a temporary outage:

- The static website still loads from Netlify.
- Dynamic public data may show loading/error states.
- Admin features will not work until Supabase is back.
- Do not try to edit data during a backend outage.

If Supabase permanently stops serving or you decide to migrate away:

- You need database and storage backups.
- Public views alone are not enough because they intentionally hide private/admin data.

Recommended backup package:

- SQL export or CSV/JSON export of `admin_profiles`.
- SQL export or CSV/JSON export of `site_settings`.
- SQL export or CSV/JSON export of `support_categories`.
- SQL export or CSV/JSON export of `fund_types`.
- SQL export or CSV/JSON export of `cases`.
- SQL export or CSV/JSON export of `case_images`.
- SQL export or CSV/JSON export of `mentorship_testimonials`.
- Storage export of `case-images`.
- Storage export of `testimonial-images`.
- Storage export of `payment-qr-images`.
- Storage export of `site-setting-images`.
- The latest committed `supabase/schema.sql`.
- The latest committed frontend code.

Auth note:

- Supabase Auth password hashes should not be treated as portable app data.
- If migrating to Firebase, create Firebase Auth users and ask admins to set/reset passwords.
- Owner/admin roles should be recreated from `admin_profiles`.

Minimum monthly backup habit:

1. Export private database tables, not only public views.
2. Download Storage bucket contents.
3. Keep the export outside the repo, for example in a private drive.
4. Never upload service role keys or `.env` to GitHub.

## 14. Firebase Migration Map

If moving to Firebase, use this mapping.

Supabase Auth -> Firebase Auth:

- Admin login becomes Firebase Auth email/password login.
- Owner/admin authorization should use Firestore admin profile documents and preferably Firebase custom claims for server-side checks.

Supabase Postgres -> Firestore:

- `admin_profiles` -> `adminProfiles/{uid}`.
- `site_settings` -> `siteSettings/{settingKey}`.
- `support_categories` -> `supportCategories/{id}`.
- `fund_types` -> `fundTypes/{id}`.
- `cases` -> `cases/{caseNumber}`.
- `case_images` -> `cases/{caseNumber}/images/{displayOrder}` or `caseImages/{id}`.
- `mentorship_testimonials` -> `mentorshipTestimonials/{testimonialId}`.

Supabase Storage -> Firebase Storage:

- `case-images` -> `case-images/...`.
- `testimonial-images` -> `testimonial-images/...`.
- `payment-qr-images` -> `payment-qr-images/...`.
- `site-setting-images` -> `site-setting-images/...`.

Supabase public views -> Firebase public-safe collections or Cloud Function projections:

- Option 1: Store private data in `cases`, then compute public stats in the frontend from only public-safe case documents. This is simpler but requires very careful Firestore rules.
- Option 2: Use Cloud Functions to maintain public-safe collections:
  - `publicCaseLedger`.
  - `publicCaseStories`.
  - `publicMentorshipTestimonials`.
  This is safer and closer to the current Supabase view model.

Supabase Edge Function -> Firebase Cloud Functions:

- `invite-admin` becomes an owner-only callable/HTTPS Cloud Function.
- Storage cleanup on case/testimonial deletion should be a Cloud Function or explicit frontend admin action.

Security rules:

- Public users should read only public-safe collections and public Storage objects.
- Admin users should read/write private collections only after role verification.
- Owner-only actions should require `role = owner`.
- Private beneficiary fields must never be readable by unauthenticated users.

## 16. Firebase Migration Reality Check

The prompt above is enough to ask Codex to start the migration, but Codex can only migrate the data if it can access exported data.

If Supabase is completely inaccessible and there is no backup:

- Codex can migrate the code structure.
- Codex cannot recover private case rows or Storage images that only existed in Supabase.

So the best insurance is not only `supabase/schema.sql`; it is regular exports of both:

- Database rows.
- Storage objects.

## 17. Project Rules to Preserve

Always preserve these rules across future changes:

- Public site must not expose beneficiary name, phone, exact private address, private notes, payment details, donor names, IDs, or admin-only fields.
- Admin panel must require login.
- Owner-only admin management must stay owner-only.
- Case numbers should stay stable and human-readable.
- Month entry should be normalized to avoid `Jun 2026` and `June 2026` duplicates.
- Public stories require explicit publish flag.
- Testimonials require consent.
- Uploaded images must be public-safe.
- Replacing QR images should remove the older uploaded QR object where possible.
- Deleting case/testimonial records should also clean their Storage images where possible.
